# Recipe matrix — adapter snippets per stack

One row per common test-runner. Each row gives the 6-line evidence-collection snippet, the matching adapter CLI, the `--validation-map` requirement (if any), and a link to a working example workflow.

All snippets below focus on evidence collection and work with either:
- **Tier 1** reusable workflow (`uses: .../workflows/readiness.yml@<sha>`), or
- **Tier 3** in-repo CLI execution (`uvx --from release-readiness-core release-readiness-evaluate ...`).

If you're adopting as a third-party consumer and want to avoid cross-repo reusable workflow access dependencies, prefer Tier 3.

## pytest + JUnit

```yaml
- run: pytest --junit-xml=test-results.xml
- run: |
    junit-to-readiness \
      --input test-results.xml \
      --output evidence/e2e.json \
      --validation-map ops/release-readiness/validation_map.yaml
```

Then call the reusable workflow:

```yaml
jobs:
  readiness:
    needs: collect-evidence
    uses: psuthar/release-readiness-core/.github/workflows/readiness.yml@<sha>
    with:
      package-ref: <sha>
      e2e-results: evidence/e2e.json
    secrets: inherit
```

Working example: [`examples/python-pytest/.github/workflows/release-readiness.yml`](../../examples/python-pytest/.github/workflows/release-readiness.yml).

## Go test (smoke + e2e split)

Use `go test -run` to split smoke vs. e2e by test-name convention (`TestSmoke...` and `TestE2E...`). Each step writes its own evidence file based on the step outcome:

```yaml
- id: smoke
  continue-on-error: true
  run: go test ./internal/app -run '^TestSmoke' -count=1

- id: e2e
  continue-on-error: true
  run: go test ./internal/app -run '^TestE2E' -count=1

- name: Write smoke and e2e evidence
  run: |
    mkdir -p evidence
    if [ "${{ steps.smoke.outcome }}" = "success" ]; then
      echo '{"status":"passed","passed":true,"api_health":true}' > evidence/smoke.json
    else
      echo '{"status":"failed","passed":false,"failed_count":1,"total_count":1,"failures":[{"title":"smoke tests failed"}]}' > evidence/smoke.json
    fi
    if [ "${{ steps.e2e.outcome }}" = "success" ]; then
      echo '{"status":"passed","failed_count":0,"total_count":1,"retries":0,"failures":[],"validations":{"todo_crud":true}}' > evidence/e2e.json
    else
      echo '{"status":"failed","failed_count":1,"total_count":1,"retries":0,"failures":[{"title":"e2e failed"}],"validations":{"todo_crud":false}}' > evidence/e2e.json
    fi
```

`continue-on-error: true` is load-bearing: it lets the readiness evaluator still receive evidence on test failure (so it can render `BLOCK` with diagnostics) instead of the workflow short-circuiting before evidence is written.

Generated by `release-readiness-init --stack go` (smoke-only variant). The smoke + e2e split shown above is what the [`release-readiness-sample-app`](https://github.com/psuthar/release-readiness-sample-app) uses end-to-end — see [its workflow](https://github.com/psuthar/release-readiness-sample-app/blob/main/.github/workflows/release-readiness.yml) for a complete, copy-pasteable Tier-3 worked example (also covered in [`docs/how-to/9-adoption-tiers.md`](9-adoption-tiers.md#tier-3-in-github-actions-complete-worked-example)).

## Go cover + LCOV

```yaml
- run: go test ./... -count=1 -coverprofile=coverage.out
- run: |
    go install github.com/jandelgado/gcov2lcov@latest
    mkdir -p coverage
    gcov2lcov -infile=coverage.out -outfile=coverage/lcov.info
- run: |
    lcov-to-readiness \
      --input coverage/lcov.info \
      --output evidence/coverage.json \
      --baseline-percent 85
```

Generated by `release-readiness-init --stack go-coverage`. `--baseline-percent` is the threshold below which `coverage_regression` fires.

## When you don't need a validation map

The validation map is only required when you want **per-validation-key satisfaction** (e.g., `auth_login: true` in the e2e shape). Without it, the adapter still emits valid `e2e.json` (counts, failures, status) — and you can use `infer_validations_when_pass` in `config.yaml` to mark validation keys satisfied when the suite passes overall.

See [`docs/how-to/1-map-evidence.md`](1-map-evidence.md) for the full evidence-channel walkthrough.

## Complete with Tier-3 CLI execution (third-party-safe)

After your stack-specific evidence steps, run readiness directly from your own workflow. Pin the version with a single workflow-level env var so smoke/e2e/readiness jobs all use the same release:

```yaml
env:
  RELEASE_READINESS_CORE_VERSION: "0.3.4"

# ...later, in the readiness job:
- uses: astral-sh/setup-uv@v6
- run: |
    uvx --from "release-readiness-core==${{ env.RELEASE_READINESS_CORE_VERSION }}" \
      release-readiness-evaluate \
        --repo-root . \
        --config ops/release-readiness/config.yaml \
        --smoke-results evidence/smoke.json \
        --e2e-results evidence/e2e.json \
        --coverage evidence/coverage.json \
        --enforcement-mode block_only
```

This avoids requiring access to reusable workflows in the source repo. Prefer the env-var pin (one place to bump) over inlining `release-readiness-core==X.Y.Z` literally in each step.

**Install alternative**: if you don't want a `uv` dependency, the package installs cleanly via `pip` in a venv:

```yaml
- uses: actions/setup-python@v5
  with:
    python-version: "3.12"
- run: |
    python -m venv .venv
    .venv/bin/python -m pip install -U pip
    .venv/bin/pip install "release-readiness-core==${{ env.RELEASE_READINESS_CORE_VERSION }}"
- run: .venv/bin/release-readiness-evaluate ...
```

Both produce identical output. Pick whichever your project already uses.

For a complete Tier-3 GitHub Actions workflow — including PR-vs-push base-ref handling, Check publishing, sticky comment, and the BLOCK enforcement step — see [`docs/how-to/9-adoption-tiers.md`](9-adoption-tiers.md#tier-3-in-github-actions-complete-worked-example).
